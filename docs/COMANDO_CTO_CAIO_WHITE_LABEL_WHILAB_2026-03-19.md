## Leitura Obrigatoria

Este documento e o manual operacional para qualquer engenheiro que assumir o WhiLab daqui para frente.

Nao trate este repositorio como extensao da Finnextho.
Nao trate este repositorio como software de pecuaria pronto para qualquer cliente.
Nao trate este repositorio como template cru pronto para vender sem criterio.

Trate o WhiLab como um ativo de produto em recuperacao e productizacao.

## Missao

A missao nao e apenas fazer a base funcionar.

A missao e:

1. estabilizar a plataforma
2. separar core de demo vertical
3. limpar marca e narrativa
4. elevar o produto para padrao medio vendavel
5. endurecer auth, config e docs
6. empacotar a oferta para venda honesta

O WhiLab deve ser vendido como:

- base SaaS white-label com IA
- rebranding rapido
- front-end e back-end inclusos
- autenticacao
- dashboard
- integracao com Supabase

A pecuaria pode continuar existindo, mas apenas como demo vertical.

Ela nao pode continuar definindo a identidade publica do produto.

Tudo que hoje estiver preso a:

- animais
- rebanho
- manejo
- leite
- producao
- vendas como narrativa rural

precisa ser reinterpretado em uma estrutura de:

- core
- vertical demo
- camada de customizacao

## Prioridades Innegociaveis

### 1. White-label acima do nicho

Toda decisao deve responder esta pergunta:

`isso aumenta o valor do produto como base white-label ou prende mais o projeto a uma vertical?`

Se prende mais a vertical sem necessidade, a decisao esta errada.

### 2. Produto vendavel acima de demo bonita

Nao adianta fazer landing bonita se:

- auth falha
- setup quebra
- env esta confuso
- Supabase nao fecha
- docs nao sustentam entrega

### 3. Clareza comercial acima de promessa larga

O comprador precisa entender:

- o que recebe
- o que precisa configurar
- o que e demo
- o que e white-label
- o que depende de customizacao
- o que nao entra no pacote

### 4. Rebranding acima de apego ao nome atual

Tudo deve ser construido para troca rapida de:

- nome
- logo
- cor
- dominio
- textos
- links
- nicho

## Mapa Estrutural

### Core que deve sobreviver

- shell de autenticacao
- layout principal
- dashboard base
- integracao com Supabase
- infraestrutura de API
- padroes de CRUD reaproveitaveis
- base de IA ou chat
- scripts de setup
- documentacao de deploy

### Vertical demo que deve ser isolada

- paginas de pecuaria
- rotas e entidades de rebanho
- textos rurais
- naming preso ao agro
- exemplos de dados do nicho

### Coisas que devem desaparecer da superficie publica

- residuos de Finnextho
- claims confusos de propriedade intelectual
- promessas universais
- ativos com direito indefinido
- qualquer segredo real

## Ordem Correta De Execucao

### Fase 1 - Estabilizacao tecnica

Objetivo:

- garantir base previsivel e buildavel

Entregas minimas:

- frontend build sem erro
- backend build sem erro
- auth coerente
- Supabase documentado
- env saneado

### Fase 2 - Extracao white-label

Objetivo:

- separar produto base de demo vertical

Entregas minimas:

- tokens de marca centralizados
- layout premium neutro
- textos publicos reposicionados
- dominio pecuaria empurrado para demo

### Fase 3 - Productizacao

Objetivo:

- transformar o repositorio em entrega compravel

Entregas minimas:

- README honesto
- guia de setup
- guia de rebranding
- guia de deploy
- delimitacao do que entra e do que nao entra

### Fase 4 - Oferta Kiwify

Objetivo:

- vender sem mentir e sem contaminar a Finnextho

Entregas minimas:

- promessa curta
- lista do que o comprador recebe
- lista do que precisa configurar
- upsells claros
- limites claros

## Regra De Prioridade

Se houver conflito entre:

- parecer grande e ficar vendavel
- largura e clareza
- demo bonita e produto de verdade

sempre escolher:

- ficar vendavel
- clareza
- produto de verdade
